# 数据服务与本地存储模块说明文档 (docs/data_fetcher.md)

本文件详述了**数据服务与本地存储模块**的最终实现方案、业务规则以及更新后的品种分类，作为该模块的最终确认依据。

---

## 二、 核心技术方案与业务逻辑

### 1. 流动性过滤机制（Liquidity Filter）
为了排除流动性不足、交易不活跃的死品种，我们设定了两个过滤门槛（两者满足其一即可）：
* **沉淀资金门槛**：主连合约 `沉淀资金（持仓量 * 最新价 * 乘数 * 做多保证金率）` $\ge$ **5,000万 RMB**。
* **日成交额门槛**：主连合约 `日成交额（成交量 * 最新价 * 乘数）` $\ge$ **1亿 RMB**。
* **交易所过滤**：剔除中国金融期货交易所（`CFFEX`）的股指及国债期货，只保留**商品期货**。

### 2. 价格与合约代码修复逻辑
* **GFEX 价格缺陷校正**：针对广期所（GFEX）等品种在盘后 `futures_fees_info` 接口返回价格为 `0.0` 的问题，系统会自动在筛选阶段调用 `ak.futures_main_sina` 查询主力连续合约（如 `LC0`, `SI0`）的最新实际价格进行流动性重算，避免漏掉碳酸锂和工业硅等大品种。
* **CZCE 合约代码年份扩充**：郑商所（CZCE）的实际主连合约代码在交易所和费率表中均以 3 位数字展示（如 `SR609`, `AP610`），但新浪 API 要求 4 位数字年份（如 `SR2609`, `AP2610`）。系统会自动进行扩充转换，同时保证在本地 SQLite 数据库中仍以用户侧习惯的 3 位小写字符（如 `sr609`, `ap610`）存储，以维持与元数据表的外键一致性。
* **主连/实际合约判定**：将简单的 `endswith('0')` 判定机制修改为与 `symbol + '0'` 进行字符串匹配（如 `AP0` 为主连，`AP610` 为 10 月合约），彻底解决了十月份合约（以 `10` 结尾，如 `AP610`）被误判为主连合约而无法正常扩充年份的 Bug。

### 3. K线级别与 2H 合成逻辑
* **日度 K线**：直接拉取新浪主力连续合约 `0`（如 `RB0`）及当前对应实际主力合约（如 `rb2610`）。
* **2小时（2H）K线**：拉取 60分钟 级别原始 K线，根据每个品种的交易时间类型进行对齐合并（通过时间戳进行绝对对齐，防止因节假日数据缺失导致错位）：
  * **Type 1/2/3**（日盘及23点/01点夜盘品种）：
    * 上午第一时段：`10:00:00` + `11:15:00` $\rightarrow$ 合并为 **`11:15:00`** 棒（对应 09:00 - 11:15 交易时段）。
    * 下午第二时段：`14:15:00` + `15:00:00` $\rightarrow$ 合并为 **`15:00:00`** 棒（对应 11:15 - 15:00 交易时段）。
    * 夜盘时段：`22:00:00` + `23:00:00` $\rightarrow$ 合并为 **`23:00:00`** 棒；`00:00:00` + `01:00:00` $\rightarrow$ 合并为 **`01:00:00`** 棒。
  * **Type 4**（02:30夜盘/黄金/白银/原油等）：
    * 日盘：`09:30:00` + `10:45:00` $\rightarrow$ **`10:45:00`** 棒；`13:45:00` + `14:45:00` $\rightarrow$ **`14:45:00`** 棒；`15:00:00` $\rightarrow$ 独立为 **`15:00:00`** 棒。
    * 夜盘：`22:00:00` + `23:00:00` $\rightarrow$ **`23:00:00`** 棒；`00:00:00` + `01:00:00` $\rightarrow$ **`01:00:00`** 棒；`02:00:00` $\rightarrow$ 独立为 **`02:00:00`** 棒。

## 三、 商品板块分类一览表 (共计 71 个活跃品种)

当前系统中 71 个活跃商品期货品种已完全归入以下 **9 个板块**：

* **包含品种**：黄大豆1号 (A), 黄大豆2号 (B), 豆粕 (M), 菜籽油 (OI), 棕榈油 (P), 花生仁 (PK), 菜籽粕 (RM), 豆油 (Y)

### 3. 新新能源 (共 3 个品种)
* **包含品种**：碳酸锂 (LC), 工业硅 (SI), 多晶硅 (PS)
